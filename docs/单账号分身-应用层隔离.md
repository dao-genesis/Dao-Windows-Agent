# 单账号多分身 · 应用层隔离（同软件多开不缠绕）

> 道并行而不相悖：RDP 会话层隔离 ≠ 应用实例隔离。本文记录真机取证的根源缺陷与根治。

## 根源缺陷（真机复现）

同账号 `dao` 两路 RDP 分身（termsrv 会话 2 / 3）先后默认启动 VS Code：

```
session 2: 11 个 Code.exe（含分身2"启动"的那次产生的新窗口）
session 3: 仅 1 个 crashpad-handler
```

分身2 的桌面上**没有任何 VS Code 窗口**——启动请求被分身1 的实例吞掉，窗口开到了错误的会话。

机理：Electron/Chromium 系（VS Code、Devin Desktop、Chrome、Edge…）的**单实例锁是
per-user、不是 per-session**——锁/命名管道落在共享的 `%APPDATA%`。同账号所有分身共享
同一用户目录，第二次启动即被转发给已存在实例。这正是"两个分身开同一软件缠在一起"的本体。

## 根治

把单实例锁的作用域从 per-user 收窄到 per-clone：每个分身用独立的
`user-data-dir`/用户配置目录（`C:\dao_clones\<clone_id>\<app>`）。

- 规格构造（纯逻辑·CI 可测）：`core/clone/app_isolation.py` → `build_clone_launch(app, clone_id)`
- guest 内落地启动器（在分身自己的会话里执行，分身号默认取当前会话 id）：

```powershell
powershell -ExecutionPolicy Bypass -File C:\dao_win\dao-clone-open.ps1 -App vscode
powershell -ExecutionPolicy Bypass -File C:\dao_win\dao-clone-open.ps1 -App devin-desktop
```

已登记软件：`vscode`、`devin-desktop`(Windsurf 同源)、`chrome`、`edge`、`freecad`。
部署链：`build_image.sh` 随盘带入 → `firstlogon.ps1` 落地 `C:\dao_win\dao-clone-open.ps1`。

## 真机验证结果（QEMU Win11 · 账号 dao · 会话 2/3）

- VS Code：两分身各 8 进程、各自 `C:\dao_clones\session-{2,3}\vscode\data`，各显各窗口。
- Devin Desktop：两分身各自实例（8/9 进程），各自 `...\devin-desktop\data`。
- 键盘输入零串扰：两分身各输入不同文本各归其窗。
- 生命周期隔离：强杀会话3 全部 Code 进程，分身1 实例与未保存文本原样存活。

## 诚实边界

- 本机制只对"有 user-data-dir/配置目录开关或环境变量"的软件零配置奏效；把单实例互斥体
  钉死在全局命名空间且无任何开关的软件，单账号零配置隔离是 Windows 固有约束
  （AGENTS.md 三节），需退回 CreateDesktop 消息级或独立账号。未登记软件
  `build_clone_launch` 如实返回 `isolatable=False`。
- 分身数上限受 guest CPU/RAM、termsrv、guacd、浏览器渲染等资源约束，非字面"无限"。
