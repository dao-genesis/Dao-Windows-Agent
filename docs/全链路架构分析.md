# 反者道之动 · 从根本底层：把整个 Windows 搬进 IDE、AI 全权接管一切软件

> 目标（重新锚定的本源）：**不是投屏**。是让每个 IDE 窗口成为一个「类虚拟机」隔离会话，
> AI 在其中全权接管、操作用户电脑一切资源与软件（KiCad/嘉立创EDA/FreeCAD 及任意软件），
> 与用户真实桌面**并行、互不干扰**；开 N 个窗口 = N 个隔离实例；单账号、零 RDP/多账号配置。
> 彻底规避传统「AI 控机 / AI 控软件 / 依赖多 RDP 虚拟机」的一切弊端。

## 一、五仓库底层现状解构（已审视）

| 仓库 | 底层驱动方式 | 关键资产 | 对本目标的意义 |
|---|---|---|---|
| **Dao-3D-Modeling-Agent**（FreeCAD） | **API/内核级**：OCCT/OCP 直连内核、`freecad_backend`(60+ ops)、`_fc_remote_server` GUI HTTP :18920、SolidWorks COM 活体桥 | 意念→模型五层闭环、反向工程(FCStd/STEP→ops)、20 平台搜索 | 证明：3D 建模软件**不靠像素**，靠脚本 API/内核直驱即可全流程 |
| **Dao-PCB-Design-Agent**（KiCad+嘉立创EDA） | 路线A **CDP 直驱** `window._EXTAPI_ROOT_`(91 命名空间官方 API)；路线B **代码化** pcbnew Python + 文件生成(.kicad_pcb/Gerber/BOM) | 冷启动底座、21 DNA 模板、原理图道四件套 | 证明：EDA 两条路都是**API/协议级**，Web 编辑器用 CDP、桌面用 pcbnew，均无屏幕自动化 |
| **windows-desktop-automation** | UIA 扫描 + 截图（0.5s），"系统只采集，AI 决策" | monitor.py / click_utils 多显示器 / smart_helper(UIA↔Vision) | GUI-only 软件的**兜底**采集层（无 API 时才用） |

| **devin-remote / dao-vsix + DAO Bridge** | 整机 `/m` 投屏 + `pc_*`/`browser_*`/`vscode_*`/`plugin_*` MCP(64 工具)、零配置隧道 | UIAutomation 控件树 pc_ui_tree、CDP 全家桶、IDE 对等 | 通用操作原语 + 隧道基座——**投屏是用户明确否定的低级形态**，但其下的 pc_*/CDP 原语可复用 |
| **ha-copilot** | WS/HTTP **确定性工具层**(~2100 工具) + 原生 agent 循环 + 帛书提示词纪律 + run_tools 编排 | 已验证「让 AI 可靠驱动一个领域」的完整配方 | **方法论母本**：领域工具 + AGENT_RULE + 组合编排，可平移到"任意软件" |

### 横向本源洞察（贯穿五仓库的同一个"道"）
**所有做得好、做得稳的路线，都不是像素自动化，而是"下沉到每个软件自己的自动化面"：**
- FreeCAD → Python API / OCCT 内核 / COM
- 嘉立创EDA → CDP `_EXTAPI_ROOT_`（91 个官方 API 命名空间）
- KiCad → pcbnew Python + 文件格式
- HA → WS/HTTP 工具层

> **反者道之动**：用户真正要的"IDE 内原生操作、规避控机弊端"，**不是把投屏做得更好、
> 不是把隐藏桌面的鼠键注入做得更准**，而是**绕过 GUI，直达软件自己的脚本/插件 API/CLI/IPC/文件格式**。
> 用户列举的每个目标软件都**天生自带一等自动化 API**，且各仓库都已在用它。GUI 像素只是"无 API 软件"的最后兜底。

## 二、"IDE 窗口 = 类虚拟机隔离会话（单账号·零 RDP）"的技术真相

Windows 单账号内实现"与用户可见桌面互不干扰的并行会话"，只有三条真实路径，**按软件类型分级降级**：

### 级别 ①（首选·覆盖 90% 价值）— API/CLI/IPC 驱动的**无头进程会话**
- 软件以**后台子进程**形态由插件后端拉起并绑定到某个 IDE 窗口的"会话"；通过其**脚本 API / 插件 API / CLI / 本地 IPC** 全程驱动。
- **天然隔离、天然并行、天然不抢焦点、天然不上用户可见桌面**——因为根本不需要可见 GUI。
- 覆盖：**FreeCAD**（FreeCADCmd/内核+:18920）、**KiCad**（pcbnew/kicad-cli headless）、**嘉立创EDA**（各会话独立 Chromium+CDP，`_EXTAPI_ROOT_`）、以及任何可脚本化软件。
- **这一级完全、彻底地实现用户的愿景**，且零 RDP、零额外账号、零 RDPWrap。用户的三大目标软件全部落在此级。

### 级别 ②（GUI-only 且需真桌面）— **独立桌面对象 / 虚拟显示器** + UIA 控件级
- 无脚本 API 的 GUI 软件：用 Win32 `CreateDesktop` 在同一会话/窗口站内开**独立桌面**，把该软件启动到此桌面（`STARTUPINFO.lpDesktop`）——**不出现在用户可见桌面**。
- 输入用**消息级** `PostMessage/SendMessage`（针对具体窗口，不依赖活动输入桌面，故不抢用户焦点）；采集用 `PrintWindow`/`BitBlt`。
- **GPU 加速渲染软件**（部分 3D/OpenGL 画布）在不可见桌面可能截黑/输入不吃 → 上 **虚拟显示器驱动**（IddCx/virtual-display-driver）造离屏物理显示器，真 GPU 合成 + Desktop Duplication 采集。
- 对标 **microsoft/UFO 的 "Picture-in-Picture 桌面"**（并行自动化不干扰用户，同源思想）。

### 级别 ③（最后兜底）— 隔离桌面 + 视觉 grounding
- 连 UIA 都拿不到的软件：截图 → OmniParser 级视觉解析 → 坐标操作，仍跑在级别②的隔离桌面内。

> **诚实的边界**：对"必须真实输入队列 + 真 GPU 且无任何 API"的少数 GUI 软件，单账号零配置隔离是 Windows 的固有约束——
> 级别②的虚拟显示器方案是当前最优解，仍可做到不占用户可见桌面；只有极端情况才需回退到现有 RDP 会话路线。
> 但用户点名的 KiCad/嘉立创EDA/FreeCAD **全部在级别①**，不触及此约束。

## 三、推荐架构：通用应用适配层（Universal App-Adapter Layer）

把 ha-copilot 验证过的「工具层 + agent 循环 + 帛书纪律 + 组合编排」配方，**从 HA 一个领域泛化成"任意软件"**，
落在 devin-remote 的插件后端 + DAO Bridge 隧道 + IDE webview 之上：

```
 IDE 窗口 (webview)  ── 每窗口一个 session_id (类虚拟机实例) ──┐
   │  对话 / agent 循环（复用 ha-copilot 本源代理）            │
   ▼                                                           │
 插件后端守护进程 (Python, 复用 vm-replica daemon 骨架)         │
   │  会话管理：create/attach/destroy·端口路由·快照            │
   ▼                                                           │
 通用应用适配层  ── App Profile 注册表（热加载·软编码）─────────┤
   │  每软件一薄片 profile = {                                  │
   │    launch/attach, 自动化面(api|cli|cdp|uia|vision),        │
   │    高层动词(open/edit/run/export...), 文件约定, 提示词片段 │
   │  }                                                        │
   ├─ 级别① API 适配器：FreeCAD(:18920/内核) · KiCad(pcbnew/cli) · 嘉立创(CDP _EXTAPI_ROOT_) · …
   ├─ 级别② 隔离桌面适配器：CreateDesktop / 虚拟显示器 + UIA(pc_ui_tree 复用)
   └─ 级别③ 视觉适配器：截图 + grounding
   │
   ▼  双层暴露（沿用 vm-replica 结论）
 REST(内核) + MCP(外壳)  ← 任意 Agent/Devin/本插件 即插即用
```

### 落地要点（皆为复用，非从零）
1. **会话模型**：每 IDE 窗口 → 一个 session_id → 一组绑定的软件实例（级别①=子进程，级别②=独立桌面）。快照/销毁复用 vm-replica 的 profile 备份思路。
2. **App Profile = 薄片**：新增软件 = 写一个 profile（声明其最佳自动化面），**不重造框架**（樸散則為器）。FreeCAD/KiCad/嘉立创 profile 内容**大部分已存在于三个仓库**，只需按统一接口收编。
3. **agent 循环 & 帛书**：直接移植 ha-copilot 的 `AGENT_RULE` + run_tools 组合 + search_tools/describe_tool 自描述，把"HA 工具"换成"当前软件 profile 暴露的动词"。
4. **暴露**：REST + MCP 双层（vm-replica 已定此结论），与 DAO Bridge `pc_*` 原语共存，IDE 内/公网隧道两端可达。

## 四、为什么这样就"规避了一切弊端"
- **规避投屏低级形态**：级别① 根本不走像素，直达 API；操作是结构化、可回放、可校验的。
- **规避多 RDP/多账号配置**：级别① 无需会话隔离硬件；级别② 用桌面对象/虚拟显示器替代 RDPWrap+建账号。
- **规避抢焦点/干扰用户**：级别① 无 GUI；级别② 消息级输入 + 独立桌面/离屏显示器，用户照常用自己桌面。
- **多实例**：session_id 天然支持开 N 个 IDE 窗口 = N 个隔离实例，各自软件互不串。
- **通用**：任意软件接入 = 加一个 profile；有 API 走①、有 GUI 无 API 走②、都没有走③。
