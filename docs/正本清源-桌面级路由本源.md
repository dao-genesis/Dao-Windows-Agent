## 一、本源（先记死）

**IDE 装上本插件 → 面板里直接就是整台 Windows 的桌面本体**——整块桌面、整套 GUI、整套图形、
整套操作、所有软件、所有后台、所有模块，**与用户在自己 Windows 上看到、操作的完全一致**。
在 IDE 面板内即可像坐在那台 Windows 前一样操作一切。

- **对标**：Windows 原生**多 RDP**——一台 Windows、一个账号，可开多路远程桌面会话，
  每路是一个**并行、互不干扰**的完整桌面。我们把"每一路会话"路由进"每一个 IDE 窗口的面板"。
- **每开一个 IDE 窗口 = 一路独立的完整 Windows 桌面**（≈ 多 RDP 的一路会话 / 一个分身账号），
  与用户真实桌面、与其它 IDE 窗口**并行且互不相悖**。
- **既服务用户，也服务 Agent**：Agent 不再"在自己的 Linux VM 上操作那台 Windows VM"，
  而是**经 IDE 面板内的这路桌面**去操作，达到"操作自己虚拟机"般的全量、可记录、可回放。

### 关键分辨：不是"投屏"，是"桌面级会话路由"
用户明确否定的"投屏"= 单纯把像素流截图/推流过来（无真实会话、交互残缺、非原生）。
**新本源要的**是走 **RDP/RemoteApp 这类原生远程桌面协议**（或更高级等效路线）把一路**真实、独立、
可交互的桌面会话**送进 IDE 面板：有真实输入队列、真实 GPU 合成、真实窗口栈，与本地体验一致。
—— 即"多 RDP 的一路"直接嵌进 IDE，而非截图流。

## 三、技术路线调研（选型给后续 Agent，附实证锚点）

### 路线 A（首选）· RDP 会话 + HTML5 客户端嵌 IDE 面板
- **单账号多路并行**：`stascorp/rdpwrap` 在任意 Windows 版本上开启**并发 RDP + 同一账号多会话**
  （官方称最多约 15 路，实际受硬件/OS 限制）。这正是"一台机、一个账号、多路独立桌面"的原生实现。
  - 关键坑（已知）：需在组策略把"将 RDP 用户限制到单独会话"设为**启用**，否则同用户重连会不断新建桌面；
    部分软件（如 VSCode）单实例会抢在初始会话开窗——属**应用单实例**问题，非 RDP 问题，需按 app 规避。
- **把桌面送进 IDE 面板**：VSCode Webview 本质是**沙箱 iframe**，可跑 `<canvas>` + WebSocket。
  用 **Apache Guacamole**（`guacd` 代理 RDP → Guacamole 协议 → `guacamole-common-js` 在 canvas 上
  渲染 + 键鼠抽象）即可把一路 RDP 桌面渲染进 Webview。`guacd` 默认 4822，Web 端经 WebSocket/HTTP 隧道接入。
  - 备选前端：`FreeRDP`（原生/WASM）、noVNC（若走 VNC）。Guacamole 生态最成熟、文档最全，**首选**。
- **每个 IDE 窗口 = 一路会话**：插件按工作区哈希（现成 `ide_<hash>` 机制）稳定映射到**一路 RDP 会话**，
  首开即经控制面（桥）拉起/连接该路会话，渲染进本窗口面板。
- **落点**：`冷启动 VM 内`：装 rdpwrap + guacd（或在宿主 Linux 跑 guacd 连 guest RDP）；
  插件 Webview 装 `guacamole-common-js`。**这是端到端闭环的主干**。

### 路线 B（并行/兜底）· 每会话虚拟显示器 + Desktop Duplication 推流
- `CreateDesktop` + IddCx 虚拟显示器离屏合成 + Desktop Duplication 采帧 → 编码 → WebSocket → 面板 canvas。
- 更贴合"单账号零配置、不建 RDP 会话"，但**工程量大**（要写虚拟显示器驱动、编码管线），
  且交互/输入需自建。作为**路线 A 受限时的兜底**或**特定 GPU 软件**补充。

### 路线 C（辅助层，非前端）· 软件级自动化（旧级别①②③）
- 保留为**控制面能力**：Agent 需要"精确、可脚本化、无头"操作时走桥 `/api/*`（exec/file/proc、
  profile 动词、UIA、PrintWindow 取证）。它**辅助**桌面路由（做编排、校验、批处理），不再是主前端。

**选型结论**：主干走 **路线 A（RDPWrap 多会话 + Guacamole 嵌 Webview）**；路线 B 兜底；路线 C 作控制面。

## 四、目标前端形态（后续 Agent 的实现靶心）

```
VSCode 窗口 A ──► 面板 = 一整块 Windows 桌面(会话 S_A)  ┐
VSCode 窗口 B ──► 面板 = 一整块 Windows 桌面(会话 S_B)  ├─ 并行·互不干扰·单账号
用户真实桌面  ──► 原桌面(会话 S_0)                      ┘
      ▲                    ▲
      │  Guacamole canvas   │  控制面(桥/api) 做会话编排·健康·脚本化辅助
      └── RDP(rdpwrap 多会话) ┘
```
- 面板首帧即完整桌面，鼠键直达，所见即所得；关掉窗口即断开该路会话（可选保活）。
- Agent 与用户共用同一前端语义：都在"IDE 面板里的这路桌面"上操作。

## 七、扩展本源（第二轮锚定 · 2026-07-08 · 反者道之动）

> 用户第二轮明确：本插件**本质上就是 Windows 多 RDP 的产品化**——底层技术力与官方
> 原生远程桌面**几乎完全一致、各处皆可复用**。因此不止"渲染一路桌面进面板"，而是要
> **彻底对齐并纳入整套 Windows 远程桌面体系**，且与 `devin-remote` 仓库既有成果合流。

### 7.1 一句话本源（升级版）
把"用户 Windows 上的任意一路 RDP 会话"当成**一台类虚拟机**送进 IDE 面板：Agent 经面板
操作它，效果与 Devin 操作自己的云 VM **1:1 等价**；且**不限账号数量**——主账号、子账号、
新建账号皆可各成一路，**并行互不干扰**；更进一步，即便用户只有**一个** Windows 账号，
也能经"本地全量复刻"造出类原生虚拟机、与真机同步又互不相悖。

### 7.2 四条能力主线（全部对齐官方原生，最大化复用）
1. **多 RDP 会话路由**（已跑通·PR#18/#19）：RDP→guacd→WS 隧道→IDE 面板 canvas，一窗一路。
2. **多账号 / 账号生命周期**：`New-LocalUser` + `Remote Desktop Users` + RDPWrap；
   创建 / 管理 / 销毁子账号，每账号一路独立完整桌面，全在插件内呈现。
3. **零配置内网穿透**：复用 `devin-remote` 的 `dao-bridge`（cloudflared + ntfy mesh 兜底、
   自愈自反注入），让**任意版本 Windows**（家庭/专业/企业/教育）开箱即得公网可达的多 RDP 通道，
   不需公网 IP / 不改路由器 —— 把"市面成熟但配置繁琐"的多 RDP 内穿一键化、原生化。
4. **本地全量复刻（VM-Replica）**：`robocopy` profile + `HKCU` 导出做账号级快照/克隆，
   单账号也能派生"类虚拟机分身"，与主账号文件/进程/HKCU 隔离、系统盘共享，秒级创建、断线保活。

### 7.3 复用锚点 · `devin-remote/cloud/vm-replica`（既有成果，直接合流，勿重造）
该目录已沉淀完整可复用内核与工程资产，**下一 Agent 必读并接续**：
- `02_Windows多RDP基础.md`：多会话三方案（Server 原生 / **RDPWrap**（Win10/11 首选）/ termsrv 补丁）、
  账号生命周期 PS、回环 RDP（`127.0.0.2` 绕自连限制）、会话内自登、隔离性评估表。
- `05_架构探讨.md`：**基底 = Python 常驻守护进程；操作层 = REST(内核)+MCP(外壳)** 的正规化结论；
  `vm_host_daemon`（账号编排/端口路由）+ 各会话内 `vm_inner_agent`（截图/输入直作用于本会话桌面）。
- `HANDOFF_NEXT_AGENT.md`：**23 个 MCP 工具**（vm_create/attach/destroy/list/exec/screenshot/
  click/type/key/...）已实测，14/14 原语与本体 VM 1:1 等价；含 5 类离屏 RDP 真实缺陷根因与修复

### 7.4 两半合一（本仓 × devin-remote）
- **本仓 `Dao-Windows-Agent`**：主攻**前端形态**——把一路真实桌面会话渲染进 IDE 面板（路线 A），
  以及冷启动开发底座（QEMU/KVM Win11）。
- **`devin-remote` vm-replica**：主攻**内核**——多账号类虚拟机的 host 守护 + inner agent + MCP 工具集，
  及零配置内穿（dao-bridge）。
- **合流方向**：本仓面板经隧道/桥连到 vm-replica 内核暴露的每一路会话；面板是"看得见的手"，
  内核是"操作层与账号编排"。二者对齐同一套原语语义（照抄 Devin computer/exec 工具），**一次实现处处可用**。
