# 三插件融合架构 · 反向推演（自最高层次模块推回本源）

> 道生一，一生二，二生三，三生万物。本文从用户描绘的**最高层次模块**（通用调度层）
> 反向推演出底层目标、架构与需求，把三插件融合的每一层理清楚，作为后续演化的正典。

## 〇、最高层次（终局）：通用调度层

终局形态：AI 在 IDE 内经桌面级路由全权操作用户整台 Windows，编程与整机操作融为一体；
当用户需求触及某专精领域（3D 建模 → FreeCAD，PCB → KiCad/嘉立创 EDA，智能家居 → HA），
AI **自主**在整机桌面下打开对应软件、调用对应领域模块高效执行，工程完成后关闭并交接给
下一环节——多领域多环节接续联动，不再是专一领域的孤岛。

由终局反推，底层必须具备四个不变量：

1. **桌面级路由**：整台 Windows 桌面本体（RDP 原生会话，非截图推流）进 IDE 面板，
   用户所见即 AI 所为，且与用户真实桌面并行不悖（每 IDE 窗口=一路隔离会话）。
2. **统一能力面**：任何软件的能力以 profile（薄片）声明为动词，经桥 `/api/*` 与 @ 调度
   统一唤起——AI 择路只看能力清单，不关心软件是内置还是子插件收编（樸散則為器）。
3. **模式切换层**：提示词与工具面按场景整体替换/裁剪——这是"编程 AI 集合体"能演化为
   "3D 建模 AI 集合体 / 智能家居 AI 集合体"的枢纽。
4. **提示词隔离替换**：底层系统提示可被整体替换（Proxy Pro 既有能力），使每种模式
   注入各自的本源纪律而互不污染。

## 一、三插件各自的本源与融合定位

| 插件 | 仓库 | 本源能力 | 融合定位 |
|---|---|---|---|
| Devin Desktop VSIX（dao-desktop） | windsurf-assistant `plugins/dao-desktop` | 官方 IDE 全量降维成插件：Cascade 面板 + language_server RPC 全家桶（编程工具面） | **AI 交互基底**（已 vendor 进本仓 `ide/vscode/dao-ai-base`） |
| Proxy Pro（dao-proxy-pro） | windsurf-assistant `plugins/dao-proxy-pro` | 底层提示词隔离替换（帛书道德模式）+ 外接第三方模型路由 | **提示词替换引擎**（SP 引擎薄片已 vendor 进本仓 `ide/vscode/dao-proxy-pro`, 读 `~/.dao/mode.json` 契约联动） |
| Windows 插件（本仓） | Dao-Windows-Agent | 桌面级路由（Guacamole canvas ← guacd ← RDP）+ 机控桥 `/api/*` + profile/@ 调度 + 子插件收编 | **整机操作本体 + 融合宿主** |

融合方式不是把三份代码搅在一起，而是**分层各司其职**：

```
IDE 窗口（=一路隔离 Windows 会话）
├── 桌面路由面板（daoWin.openDesktop）    ← 本仓：真实桌面所见即所得
├── Cascade AI 面板（daoWin.cascade*）    ← dao-ai-base：Devin Desktop 同源交互
├── 模式切换（daoWin.switchMode）         ← 本仓：模式态 → 桥 → ~/.dao/mode.json
│      └─ Proxy Pro 读契约 → 替换系统提示词/模型路由
└── 机控桥 /api/*（会话/动词/@ 调度/模式） ← 本仓：Agent 的精确操作后端
```

## 二、模式切换层（本次落地）

一个模式 = **提示词覆盖 + 工具面裁剪**（`core/agent/modes.py`）：

| 模式 | tool_policy | replace_official | 语义 |
|---|---|---|---|
| `primary` 主模式 | all | 否 | 帛书纪律 + 官方编程工具全集 + 整机操作为辅（日常默认） |
| `coding` 纯编程 | none | 否 | 官方 Devin Desktop 原貌，机控面整体关闭 |
| `windows` Windows 全接管 | all | 是 | 整机桌面级路由为主战场，编程面退辅 |
| `domain:<app_id>` 专精领域 | domain | 是 | 只保留该领域工作层 + 整机通用层 |

要点：

- **domain 模式不是硬编码清单**：从 ProfileRegistry 领域层画像现算。新增一个领域
  profile（或收编一个领域子插件），就自动多出一个专精模式——框架不动。
- **契约文件** `~/.dao/mode.json`：当前模式持久化为 JSON（mode/name/tool_policy/
  replace_official/updated），Proxy Pro、dao-desktop 等同装插件读它联动替换各自的
  提示词与工具配置——跨插件融合不靠 RPC 耦合，靠一份共享契约（无为而无不为）。
- **桥端点**：`GET /api/mode.list`、`GET /api/mode.get`、`POST /api/mode.set`；
  `/api/session.prompt` 与 `/api/capabilities` 已模式感知（提示词=模式覆盖+帛书纪律+
  已开软件纪律；能力清单按工具面策略裁剪）。
- **IDE 入口**：命令 `daoWin.switchMode`（QuickPick）+ 状态栏模式指示。

## 三、四大板块推进次序（用户既定）

1. **板块一 · Windows 专精模块**（本仓主线）：桌面级路由、机控桥、多账号并行会话、
   AI GUI 高效操作（收编 Devin-Remote 的 pc_*/browser_* 语义面）→ 推进到极致。
2. **板块二 · 基础设施**：冷启动（coldstart/up.sh 幂等编排 QEMU/KVM 真 Windows）、
   guest 内 VS Code + 融合插件部署、guacd/tunnel、CI。
3. **板块三 · 专精领域模块**（基建先行）：FreeCAD/KiCad/嘉立创/HA 各仓独立演化，
   本仓只保证 profile 收编接口 + domain 模式自动生成，适当拿现有进度做调度测试。
4. **板块四 · 通用调度层**（基建先行）：@ 调度 + 能力清单已是雏形；终局的"AI 自主
   择模块、跨领域接续联动"在板块三成熟后开发。

## 四、与 Devin-Remote AI GUI 体系的关系

Devin-Remote（dao-vsix/dao-bridge）已有整机操作语义面：`pc_*`（键鼠/文件/UI 树）、
`browser_*`（CDP 级浏览器直驱）、`vscode_*`（IDE 对等）。融合原则：

- 桌面级路由（本仓）负责**所见即所得的前端主形态**；
- AI 高效操作走**语义优先**（UIA/CDP/API 动词），规避低效的截图+点击循环——
  本仓 `core/gui/agentctl`（vendored）与桥动词即此层；
- Devin-Remote 的语义面按 profile/子插件收编进能力清单，不重造。
