# 调研 · 任意 Windows 环境接入 — 方法全景 / 多 RDP 真相 / 「装即用」边界 / 分层架构

> 道法自然 · 取之君子，用之如泥沙。本文**只调研、不实践**（用户明确「三思而后行」）：
> 系统盘点市面**所有**方法，回答「一个插件装上就完美接入用户电脑、道并行不相悖」在
> 各版本 Windows / 各权限 / 各网络状态下**到底能到什么程度、哪里是硬约束、哪里有根本解**，
> 并据此给出**推荐分层架构 + 安装首运流程 + 产品分级 + 缺陷风险清单 + 里程碑**。
>
> 前序结论见 [`正本清源-桌面级路由本源.md`](正本清源-桌面级路由本源.md) 与 [`研究归档/vm-replica_02_Windows多RDP基础.md`](研究归档/vm-replica_02_Windows多RDP基础.md)。本文在其上做**全景横向对比 + 装即用问答**。

## 〇、唯一本源目标（先钉死，防跑偏）

**用户装上插件 → 就能完美接入自己这台 Windows → 在里面并行开多个第三方软件/实例做操作 →
每个实例输入不串线、桌面状态独立、底层数据按需共享 → 且不打扰用户自己正在用的桌面。**

拆成四条可度量的硬指标：
1. **接入**：AI/面板能看见并操作这台机器的真实 GUI（不是残缺的按钮面板，也不是纯像素投屏）。
2. **并行多实例**：同机能同时跑 N 份同一款第三方软件，各自独立。
3. **隔离**：输入队列独立（不抢焦点/不串键）、桌面/窗口栈独立、`HKCU`/配置目录独立；**数据可选共享**。
4. **不相悖**：用户自己的桌面、用户原有的远程桌面连接功能，**都照常可用、无损**。

> 一句话难点：**Windows 的「一个交互式登录会话只有一张可见桌面 + 一条真实输入队列」是内核级约束**。
> 「同机多路真正并行的完整桌面」在原生层面只有两条路——**多会话（多账号/多 RDP session）**，
> 或**多虚拟机**。其余所有技巧都是在这条约束周围「取巧」，各有能力天花板。这就是为什么没有
> 「一个方法通吃」——**通吃的是「分层编排 + 能力探测 + 自动降级」，不是某单一技术**。

## 一、方法全景（市面所有路，逐条拆：原理 / 版本 / 权限 / 重启 / 隔离真伪 / 成本 / 风险）

### A 类 · 原生多会话（真隔离，但有配置/版本门槛）

#### A1. Windows Server 原生 RDS 多会话
- **原理**：Server 默认允许 2 个管理会话；装 RD Session Host 角色后并发无上限。
- **版本**：仅 Server（2016/2019/2022/2025）。**桌面版 Windows 不适用**。
- **权限/重启**：需管理员装角色 + 改组策略「限制单会话=禁用」；**装 RDS 角色需重启**。
- **成本**：RDS CAL 授权（120 天宽限期可临时用，之后需买 CAL）；域环境还要 RD Licensing。
- **隔离真伪**：**真**——每会话独立完整桌面 + `HKCU` + profile；仅 `HKLM`/系统盘共享。
- **风险**：授权成本；面向企业/服务器，普通用户机器根本没有。
- **判定**：稳、真隔离，但**只覆盖极小一部分设备（服务器）**，不是通用解。

#### A2. RDP Wrapper（桌面版开并发/同账号多会话的事实标准）
- **原理**：`rdpwrap.dll` 注入 `termservice`，**运行时内存补丁** `termsrv.dll`（绕单会话限制/许可 hook），
  **不改磁盘文件**。`sebaxakerhtc` 活跃 fork 的 `rdpwrap.ini` 跟进新版本。
- **版本**：**Home/Pro/Edu/Enterprise 全覆盖**（这正是它的价值——给本无 RDP host 的家庭版补上）。
- **权限/重启**：需管理员装（`install.bat` 注册服务补丁）；**通常无需重启**（重启 TermService 即可），
  但装完首次建议重启以稳。
- **成本**：开源免费；维护成本高（见风险）。
- **隔离真伪**：**真**——等价 A1 的会话级隔离，单账号也能本地+远程多会话并行（官方称约 15 路）。
- **风险（关键·必须诚实）**：
  - **供应链**：第三方 DLL 注入系统服务；来源可信度、二进制签名、被投毒风险需自评。
  - **版本脆性**：Windows Update 换 `termsrv.dll` 版本 → `ini` 偏移不匹配 → **RDP 直接失效**，需等社区更新 `ini` 或跑 autoupdate。这是**持续运维负担**，不是一次装好就永久。
  - **杀软**：Defender/第三方 AV 常报毒/拦截注入，需加排除项（又需管理员 + 用户信任）。
  - **法律/授权**：绕过微软会话许可策略，商业分发存在合规争议。
- **判定**：**能力最强的「让普通机器变多会话」的现成路**，但**供应链+更新脆性+杀软**三座大山让它
  **不适合作为「装即用、免维护」产品的默认强依赖**——只能作为**用户显式知情同意后的「增强档」可选项**。

#### A3. 直接补丁 termsrv.dll（改磁盘）
- 每次系统更新即失效、需重补，改磁盘文件更脏。**仅了解，不采用。**

#### A4. 多本地账号（配 A1/A2 使用）
- `New-LocalUser` + 加入 `Remote Desktop Users`。**账号本身不等于隔离**——
  **必须配合会话机制（RDS/RDPWrap）才有真并行**；否则只是「切换用户」（同一时刻仍只有一张可见桌面）。
- **诚实区分**：「有多个账号」或「多条 RDP 连接指向同一用户桌面」**不是真并行隔离**；
  真隔离 = 每账号/每会话各有**独立的活跃交互会话**。域环境建账号可能被域策略禁止。

### B 类 · 单会话内的桌面隔离（零配置起步，但有能力天花板）

#### B1. Win32 CreateDesktop / HDESK（桌面对象隔离）
- **原理**：同一登录会话内建多个**桌面对象**（HDESK），每个有独立窗口栈；配 `PostMessage` 消息级输入
  （不抢用户焦点）+ `PrintWindow` 采集，做到「后台桌面里跑软件，不打扰用户可见桌面」。
- **版本**：全版本（Win32 原生 API）。
- **权限/重启**：**普通权限即可，免重启、免配置**——这是它最大优点，天然「装即用」。
- **隔离真伪**：**半真**——窗口/输入焦点隔离成立；但**共享同一 `HKCU`/session**，非会话级隔离。
- **硬约束（必须诚实）**：
  - **UWP/packaged app**（Win11 记事本、计算器、商店应用）激活走 `RuntimeBroker`，**常绕过 HDESK**，
    在用户可见桌面弹出 → 隔离失效。
  - **GPU/D3D 合成、独占全屏、受保护内容（DRM）窗口**：`PrintWindow` 采不到或黑屏，需虚拟显示器。
  - **全局单实例 mutex 应用**：第二个实例直接激活第一个（属应用行为，非桌面问题，需 per-app 规避）。
- **判定**：**零配置隔离的最佳起点**，覆盖大量传统 Win32 桌面软件；但对 UWP/GPU/全屏类**天花板明显**。

#### B2. 虚拟显示器（IddCx 驱动）+ Desktop Duplication
- **原理**：装 IddCx 虚拟显示器驱动 → 离屏合成一块「显示器」→ Desktop Duplication 采帧 → 编码推面板。
  解决 B1 采不到 GPU 合成画面的问题。
- **版本**：Win10 1607+ / Win11。
- **权限/重启**：**装驱动需管理员**；**通常需重启或至少驱动加载**（用户参与「允许安装驱动」弹窗）。
- **隔离真伪**：改善**采集**，但**输入/会话隔离仍受单会话约束**；且可能影响用户真实多显拓扑（需谨慎离屏）。
- **风险**：驱动签名/兼容（部分 GPU、远程显示、DCH 驱动冲突）；驱动 = 供应链 + 稳定性风险。
- **判定**：B1 的**GPU 补充**，代价是「装驱动 + 可能重启」，已不满足「纯零配置」。

### C 类 · 无头控制面（天然并行隔离，但非「完整 GUI 桌面」）

#### C1. API / CLI / CDP / 应用原生自动化接口
- **原理**：不碰桌面，直接驱动应用的程序接口——Chromium `--remote-debugging`(CDP)、VS Code CLI、
  Office COM、各类 app 的 REST/CLI。
- **版本/权限/重启**：全版本、普通权限、免重启、**零配置**、天然多实例并行、天然隔离。
- **硬约束**：**只对「有接口的应用」有效**；对「纯 GUI、无接口」的第三方软件覆盖不到「完整桌面交互」。
- **判定**：**最稳的兜底底面 + 自动化加速层**，但**不满足「整块真实桌面」本源**，只能作控制面/辅助。

#### C2. UI Automation（UIA）
- 微软官方无障碍框架，读控件树 + 程序化触发。全版本、普通权限。对 WPF/Win32/部分 UWP 有效；
  对纯自绘/Canvas/游戏无效。**辅助层**（定位、校验、取证），非前端。

#### C3. SSH / WinRM / PowerShell Remoting
- 控制面协议：exec/file/proc、脚本化。WinRM 需启用（`Enable-PSRemoting`，管理员）；OpenSSH Server 需装。
- **不提供 GUI 桌面**。作 Agent 脚本化后端，非本源前端。

### D 类 · 虚拟机 / 沙箱（最强隔离，最重成本）

| 方案 | 版本门槛 | 权限/重启 | 隔离 | 成本 | 判定 |
|---|---|---|---|---|---|
| **Windows Sandbox** | 仅 Pro/Edu/Ent（Home 无） | 需启用可选功能 + 虚拟化 + **重启** | 真·一次性 | 轻，但**不持久**（关即毁）、无 GPU 复杂场景 | 临时隔离可，不能做持久多实例 |
| **Hyper-V** | 仅 Pro/Edu/Ent（Home 无） | 管理员启用 + **重启**；需 VT-x/AMD-V | 真·完整 | 重（内存/磁盘/授权 Windows guest） | 强隔离档，非通用默认 |
| **VirtualBox/VMware** | 全版本可装 | 装 hypervisor（管理员）；与 Hyper-V 冲突需协调 | 真·完整 | 重 + 需 guest OS 授权 | 同上 |
| **QEMU/KVM（我方 coldstart）** | 跑在**我方 Linux VM**上，不占用户机 | 用户机零要求 | 真·完整 | 我方云端资源/延迟 | **终极兜底 Z**：用户机啥都不满足时，在我们自己的 VM 上跑 Win10/11 |

- **共性硬约束**：桌面版 VM 方案（Sandbox/Hyper-V）**Home 版直接没有**；都需**虚拟化开启 + 重启**；
  都要解决 **Windows guest 授权**；资源开销大。
- **判定**：VM = **真隔离的天花板**，但成本/门槛也最高。**coldstart(Z)** 因为不依赖用户真机，是
  「任意环境」的**终极保底**——用户机再差，也能在我方 VM 上给一台干净 Windows。

### E 类 · 远程桌面协议前端 / 第三方远控（送「已有会话」进面板，不解决「造会话」）

> 关键辨析：这些是**「把一路桌面画进面板」的前端/传输**，**不负责在用户机上「造出多路并行会话」**。
> 「造会话」永远回到 A/B/D 类。选它们是选**渲染与传输通道**。

| 工具 | 角色 | 是否给「多独立会话」 | 备注 |
|---|---|---|---|
| **Apache Guacamole (guacd)** | RDP/VNC/SSH → HTML5 canvas | **否**（它连的是已存在的 RDP 会话） | 生态最成熟、Webview 友好、**路线 A 前端首选** |
| **FreeRDP** | RDP 客户端（原生/WASM） | 否 | 程序化拿帧+注入输入，可作备选前端/内核 |
| **VNC / noVNC** | 帧缓冲共享 | **否**（默认共享**同一**可见桌面，非多会话） | 除非配多会话后端，否则就是「看用户桌面」 |
| **AnyDesk / RustDesk / TeamViewer** | 远控 | **否**（默认接管**用户当前**桌面，抢焦点） | RustDesk 开源可自建，但仍是「一路共享桌面」，**不是多独立会话** |
| **Parsec** | 低延迟串流（游戏级 GPU） | 部分（虚拟显示器 + 用户级会话） | GPU 体验最佳，但仍围绕单可见/虚拟会话，商业授权 |

- **诚实结论**：**没有任何一个远控/协议前端能「变出」多路真并行会话**——它们只是把「后端已经存在的
  某路桌面」的像素+输入通道搬到面板。多实例隔离能力**完全取决于后端用的是 A（多会话）还是 D（多 VM）**。
- **VNC/AnyDesk/RustDesk/TeamViewer 默认都是「共享用户当前那一张桌面」**，会抢焦点、会串扰，
  **不满足我们「不相悖 + 并行独立」的本源**——除非只做「看/协助」，不做「并行多实例」。

### F 类 · 应用级隔离（正交增益：任何路都该叠加）

- **独立配置目录**：Chromium `--user-data-dir`、VS Code `--user-data-dir/--extensions-dir`、
  各 app 的 `APPDATA` 重定向 → 同机多实例、配置互不覆盖。**普通权限、零配置、免重启。**
- **数据按需共享**：把共享数据目录 symlink/junction 进各实例的数据路径（用户明确要「底层数据可共享」）。
- **硬约束**：**全局单实例 mutex**（Office/微信/部分 IM）——第二个实例激活第一个，`--user-data-dir` 无效，
  需更强隔离（会话级 A / VM 级 D）才能真多开；**UWP** 无命令行 data-dir 概念。
- **判定**：**每条路都应叠加的「第一层零配置多开」**，成本最低、收益最广，但**吃不下强单实例/UWP**。

## 二、传统「Windows 多 RDP 体系」到底难在哪（用户核心追问，逐条量化）

用户问：「原有那种多 RDP 体系，帮用户配置那么复杂，最大难点在哪？用户使用成本有哪些？能不能适配所有版本？」

**传统自建多 RDP 的完整配置链（用户视角，家庭版为例）**：
1. 判定版本 → Home 无 RDP host，**必须装 RDP Wrapper**（下载第三方、过杀软、管理员安装）。
2. 装完对 `rdpwrap.ini` 做版本匹配（当前 `termsrv.dll` 版本对不上要手动 autoupdate）。
3. 改组策略/注册表：单会话限制、并发数、NLA。
4. 建多个本地账号 + 加 `Remote Desktop Users` + 设密码（RDP 要求非空密码）。
5. 开防火墙 3389；家用还要路由器端口映射 / 公网 IP / DDNS（否则外网连不进）。
6. 每账号配自动登录 / 计划任务拉起会话。
7. 客户端侧存凭据（`cmdkey`）、回环 `127.0.0.2` 绕自连限制、配分辨率/多显/剪贴板。
8. **持续运维**：每次 Windows Update 后验证 RDPWrap 是否还活、`ini` 是否要再更新。

**最大难点（本质）**：
- **「造出多路真会话」这一步**——它触碰微软会话许可策略，桌面版原生不给，只能靠第三方补丁，
  于是**供应链 + 更新脆性 + 杀软**三重不确定性，且**跨版本行为不一致**。
- **网络可达**——家用 NAT 后无公网，端口映射对普通用户是劝退级操作。
- **配置面广且互相牵连**（版本→补丁→策略→账号→防火墙→穿透→客户端），**任一环错则全链不通**，
  排障需要专业知识。

**用户使用成本**：管理员权限、下载可信度判断、杀软放行、组策略/注册表编辑、路由器配置、
Windows 更新后复检——**对非技术用户几乎不可完成**，对技术用户也是「装一次要半天、更新后还得修」。

**能否适配所有版本**：
- **原生 RDP host**：Pro/Edu/Ent 有（默认单会话），**Home 完全没有**，Server 有（RDS 可多会话）。
- **多会话**：桌面版**任何版本都要靠 RDPWrap**（含 Pro）；Server 靠 RDS 角色。
- 所以**「纯原生、不装第三方、跨所有版本、都能多会话」= 不存在**。这是硬约束，不是能力不足。

## 三、「安装即用」逐问必答（用户点名的每一条）

> 分档回答：**能力分「零配置即得」「需一次授权」「需装驱动/改服务/重启」三级**。诚实标注每一档。

| 问题 | 回答（诚实） |
|---|---|
| 安装需要管理员权限吗？ | **基础档不需要**（应用级隔离 F + CreateDesktop B1 + 无头 C 均普通权限）。**增强/强档需要**（RDPWrap、虚拟显示器驱动、启用 RDP/Hyper-V/Sandbox 都要管理员）。 |
| 需要用户点击授权吗？ | 基础档几乎无。装驱动会有系统「允许安装驱动」弹窗；启用无障碍/WinRM 需一次确认；杀软放行需用户点。 |
| 需要登录/配账号/选目录吗？ | 基础档零配置（自动派生 `--user-data-dir`）。多账号档才需建账号。 |
| 需要重启电脑吗？ | **基础档免重启**。RDPWrap 一般免重启（重启 TermService）；**虚拟显示器驱动/启用 Hyper-V/Sandbox/首次开 RDP 角色可能需重启**。 |
| 哪些能力不重启即可用？ | 应用级多开(F)、CreateDesktop 桌面隔离(B1)、无头 API/CLI/CDP(C)、UIA(C2)、连接已开启的原生 RDP。 |
| 哪些必须装驱动/改服务/开功能才可用？ | GPU 软件离屏(B2 虚拟显示器)、桌面版多会话(A2 RDPWrap)、VM 强隔离(D Hyper-V/Sandbox)。 |
| 原生 RDP 已开启时能无损共存吗？ | **能**。我们只是**新增会话/连接**，不改用户已有 RDP 配置；RDPWrap 也是「增加并发」不删原功能。 |
| 原生 RDP 未开启时能自动开吗？ | Pro/Edu/Ent：可（管理员 + 用户同意，改注册表 `fDenyTSConnections=0` + 开防火墙）。**Home：原生开不了，只能 RDPWrap（增强档，需知情同意）**。 |
| Home 版无原生 RDP 怎么办？ | 三条：① 走**不需要 RDP 的路**（B1 桌面隔离 + F 应用多开 + C 无头，**这就是「装即用基础档」**）；② 用户同意则 RDPWrap（增强档）；③ 都不行走 **coldstart(Z)** 我方 VM。 |
| 无管理员权限怎么办？ | 只能基础档（B1+F+C）。**RDPWrap/驱动/VM 全部不可用**——如实降级，不假装。 |
| 域加入设备怎么办？ | 组策略可能禁 RDP/禁建账号/禁装驱动/禁虚拟化；探测到域策略阻断即**降级到基础档或 Z**，并如实告知「本机受域策略限制，强隔离不可用」。 |
| 防火墙/杀软/组策略阻止怎么办？ | 防火墙：本地回环(`127.0.0.2`)+隧道多数可绕过对外端口需求；杀软拦 RDPWrap/驱动 → 提示用户放行或降级；组策略硬禁 → 降级到 Z。 |
| 睡眠/锁屏/切换用户的影响？ | 锁屏/睡眠会中断可见会话渲染；RDP 会话断开(非注销)可保活后台运行；切换用户改变活跃会话。需**保活 + 断线重连**策略（coldstart 内核已有会话保活经验）。 |
| 「装一个插件自动配好一切」哪些可行、哪些不可能保证？ | **可行**：基础档（多开+桌面隔离+无头）真能「装上即用、免重启、免配置、与用户桌面不相悖」；探测/选路/降级可全自动。**不可能保证**：在**无管理员 / Home 无原生 RDP / 域禁用 / 杀软拦截**时，把「同机多路**完整真桌面会话**」全自动配好——这触碰系统权限与微软授权策略的硬边界，**只能靠增强档（需用户知情同意）或 coldstart 兜底**。 |

**装即用的诚实结论**：
- 「**基础档**（应用多开 + CreateDesktop 桌面隔离 + 无头控制面）**能真正做到装上即用、免重启、
  免管理员、免配置、与用户原桌面并行不相悖**」——这覆盖了**大量传统第三方 Win32 软件的多实例并行**。
- 「**完整真桌面级多会话**」在桌面版**无法在「零权限零配置」下全自动获得**——它需要管理员 + 用户
  知情同意（原生开 RDP 或 RDPWrap），或退到 coldstart。**这不是我们能力问题，是 Windows 内核 +
  微软授权策略的硬约束**。**根本解不是「突破限制」，而是「分层 + 自动降级 + VM 兜底」让每种环境都有
  一条落地路径**。

## 四、结论：有没有「一个方法通吃」？

**没有单一方法通吃**（论证见上）。但**有一个通吃的「产品策略」**：

> **零配置优先 → 能力探测 → 按环境自动选路 → 用户知情同意再升级 → coldstart 终极兜底。**

- **可统一抽象的**：接入语义（看/点/键/截屏/exec/文件）、应用 profile、会话生命周期、探测/选路/降级、
  面板前端（canvas 渲染 + 输入通道）——**一次实现，处处复用**（对齐 Devin computer/exec 原语）。
- **必须按环境分层的**：「多路完整桌面」的产生机制（B1 桌面对象 / A 多会话 / D VM）——**因版本/权限而异**。

## 五、推荐分层架构（探测驱动 · 自动降级 · 已有 `core/environment.py` 骨架）

```
                 ┌──────────────────────────────────────────────┐
                 │  探测层 core/environment.py（已落地 PR#86）    │
                 │  版本家族/构建/管理员/入域/RDP开关/单会话/RDPWrap│
                 └───────────────┬──────────────────────────────┘
                                 │ 选路 desktop_strategy()
   ┌─────────────┬───────────────┼───────────────┬─────────────────┐
   ▼             ▼               ▼               ▼                 ▼
基础档①       增强档②          增强档②'         强档③            兜底Z
应用多开(F)   桌面隔离(B1)      多会话(A)        VM(D)            coldstart
+无头(C)      +虚显(B2)        RDP/RDPWrap      Hyper-V/Sandbox   我方QEMU/KVM
零配置免重启   零配置(B1)/      需管理员+同意     需管理员+重启      不占用户机
天然并行隔离   驱动需管理员      真会话隔离        最强隔离          终极保底
覆盖有接口/    覆盖多数Win32     覆盖GPU/UWP/      覆盖HKLM级/       用户机全不满足
传统软件多开   非GPU软件         强单实例真多开    驱动级隔离        时仍给干净Win
```

- **默认落点**：任何机器**先给基础档①**（立即可用、不打扰用户）；探测到可升级且用户同意 → 升②/③；
  全不满足 → Z。**每个格子都有落地路径 = 适配任意环境。**
- **应用 profile（F）横切所有档**：每款第三方软件一张薄片，声明「需要的最低隔离档 + 数据共享/隔离策略
  + 单实例规避法」。新增软件 = 写 profile，不改框架（樸散則為器）。

## 六、安装 / 首次运行流程（推荐 · 分档 · 可回滚）

1. **装插件（零权限）** → 立即可用**基础档①**：无需重启、无需管理员、无需配置。首屏即能多开第三方软件、
   桌面隔离运行、无头驱动。**用户原有远程桌面/桌面完全无损。**
2. **后台静默探测**（`core/environment.py`）：版本/权限/RDP/域/杀软 → 算出「当前可用档 / 可升级档」。
3. **按需征询升级**（**只在用户主动要更强隔离时**）：
   - 需要 GPU 软件离屏 → 提示「安装虚拟显示器驱动（需管理员/可能重启）」，用户点「同意」才装。
   - 需要完整真桌面多会话 → 提示「开启原生 RDP（Pro+）」或「安装 RDPWrap（Home，第三方组件，
     附风险说明 + 校验）」，**默认不自动装 RDPWrap**。
   - 需要强隔离 → 提示「启用 Hyper-V/Sandbox（需重启）」。
4. **全程可回滚**：每步幂等 + 记录（关虚显驱动、卸 RDPWrap、恢复 RDP 注册表、删派生账号），
   一键还原到「就像没装过」。
5. **全不满足 / 无权限 / 域禁用** → 自动走 **coldstart(Z)**，在我方 VM 给一台干净 Windows。

## 八、缺陷与风险清单（诚实 · 做前必读）

1. **RDPWrap 供应链 + 更新脆性**：第三方 DLL 注入系统服务；Windows Update 换 `termsrv.dll` 即失效需修 `ini`；
   杀软报毒；商业分发合规存疑。**结论：不作默认强依赖，仅增强档 + 用户知情同意 + 二进制校验。**
2. **驱动安装/重启**：虚拟显示器(IddCx)、Hyper-V/Sandbox 需管理员 + 可能重启 + 用户弹窗，破坏「纯装即用」。
3. **UWP / packaged app**：激活绕 HDESK、无 `--user-data-dir`，桌面对象隔离对其失效 → 需会话级(A)。
4. **GPU / 独占全屏 / 受保护内容**：`PrintWindow` 采不到，需虚拟显示器；DRM 窗口可能永远采不到。
5. **全局单实例 mutex 应用**：Office/微信/部分 IM，应用级多开无效 → 需会话级(A)或 VM(D)。
6. **无管理员权限**：只能基础档，一切需提权的路全断——如实降级。
7. **域策略 / 杀软 / 防火墙**：可硬禁 RDP/建账号/装驱动/虚拟化 → 降级到基础档或 Z。
8. **网络可达**：家用 NAT 无公网；靠零配置内穿(dao-bridge 隧道 + mesh 兜底)化解，但隧道稳定性/延迟需打磨。
9. **资源成本**：VM 档内存/磁盘/授权；coldstart 云端资源/延迟/隐私（用户数据出本机需明确同意）。
10. **数据隐私**：远程/云路径涉及用户数据流出；凭据/token 严禁入库入日志（走既有安全通道）。
11. **睡眠/锁屏/切用户**：中断渲染/改活跃会话，需保活 + 断线重连。
12. **「共享桌面」伪隔离陷阱**：VNC/AnyDesk/RustDesk/多 RDP 连同一用户 ≠ 真并行；必须坚持「每路独立活跃会话」。
