# 四大板块 · 底层解构（反者道之动）

> 反也者，道之動也。本文把四大板块反向解构到底：每块的**底层驱动力 → 底层方向 → 底层目标 → 现状 → 缺口**，
> 作为逐块全栈闭环（前端+后端+实践+验证+使用）的推演基准。现状证据均来自本 VM（QEMU Win11 guest）实机取证。

## 板块一 · Windows 全接管（整机桌面级路由）

- **底层驱动力**：用户要 AI 全权操作整台电脑，且**与用户真实桌面并行、互不干扰**——
  规避投屏/抢焦点/独占会话等旧方案弊端。
- **底层方向**：把整台 Windows 做进 IDE 插件。每个 IDE 窗口 = 一路独立完整桌面会话
  （≈ 多 RDP 的一路），面板 = 整块桌面本体（guacamole canvas → WS 隧道 → guacd → RDP），
  不是像素投屏而是原生桌面级会话路由。
- **底层目标**：用户在 IDE 面板内所见即所得；AI 的一切操作（开软件/调度/开发）用户全程可见；
  用户主桌面零打扰。
- **现状**（guest 实机取证）：VSIX 0.3.0 装入 guest VS Code，扩展激活、状态栏
  `DAO ide_<hash>` + 模式项就位；「打开桌面」「选账号开桌面」「新建 Windows 账号」命令齐备；
  桌面面板 webview（desktopHtml）+ 剪贴板双向 + 账号切换消息通路已实现。
- **缺口**：guest 内 guacd/RDP 隧道链路尚未在冷启动脚本里一键拉起（面板可开、连接待通）；
  多账号并发路由的实机压测未做。

## 板块二 · 三插件融为一体（AI 交互基底 + 提示词隔离 + 机控）

- **底层驱动力**：Devin Desktop VSIX（编程基底）、Proxy Pro（提示词隔离替换）、本仓机控
  三者若各自为政，Agent 被"一股脑工具"淹没、提示词与工具面脱节。
- **底层方向**：合三为一，以**模式**为枢纽：一个模式 = 提示词覆盖 + 工具面裁剪；
  态持久化为 `~/.dao/mode.json` 契约，同装引擎（dao-proxy-pro / dao-ai-base）读契约联动。
- **底层目标**：primary（帛书纪律+编程全集+整机为辅）/ coding（官方原貌，机控面关）/
  windows（整机主战场）/ domain:*（专精）随点随切，前后端一致生效。
- **现状**（guest 实机取证）：命令面板 11 条 DAO 命令齐活；`提示词引擎: 状态` 报
  `经藏 laozi+yinfu (7715 字) · 模式 primary · overlay`；`道化预览` 开出道化后 SP 全文；
  状态栏点击弹七模式 QuickPick，切到 `Windows 全接管` 后状态栏即时更新、桥
  `/api/mode.get` 同步返回 windows；82 项单测全绿。
- **缺口**：模式切换后已开的 Cascade/AI 交互面板不主动刷新能力面（需重开）；
  dao-ai-base 的 Cascade 会话与桥会话（session_id）尚未互认。

## 板块三 · 专精领域工作层（FreeCAD / KiCad / 嘉立创 / …）

- **底层驱动力**：重型专业软件各有原生 API/CLI/CDP（级别①），伪装 GUI 点击是下策；
  各仓（Dao-3D / Dao-PCB / ha-copilot）已有驱动成果，不重造。
- **底层方向**：每个软件 = 一份 profile 薄片（樸散則為器），领域层用 `@句柄` 唤起；
  外部子插件经 `~/.dao/subplugins/*.json` 描述符 + RPC 收编为 external 画像，与内置一视同仁。
- **底层目标**：`domain:<app>` 专精模式 = 只保留该领域动词 + 整机通用层；
  新增一个 profile/子插件自动多出一个专精模式与一路 @句柄，框架零改动。
- **现状**：kicad/freecad/jlceda/paint/notepad/system/browser 画像就位；
  `/api/route` 实测 `@kicad 导出 gerber` → domain 层 + `export_gerbers` 候选命中；
  模式 QuickPick 自动列出四个专精模式；子插件发现/收编（core/subplugin.py）+ 单测就绪。
- **现状追加**：本机真装 KiCad pcbnew / FreeCAD 0.19 级别① invoke 实跑全通（PR #35/#36）；
  子插件宿主 `bridge/subplugin_host.py` 落地——一份 spec 即起真实本地 `/invoke` 端点并写出
  描述符自动收编（首个实例：`bridge/subplugin_specs/homeassistant.json` 包 ha-copilot/hactl）。
- **缺口**：guest（Windows）内真装软件的 invoke 实跑未做；HA 实例在线端到端（真开灯）待真机。

## 板块四 · 通用调度层（终局：跨领域自主接续）

- **底层驱动力**：终局形态里用户全栈提需求（PCB 完了接 3D，3D 完了接下一环节），
  Agent 必须自主感知"有哪些领域模块可用"并择路、交接。
- **底层方向**：无 @ 的目标落整机通用层；`@句柄` 唤起领域层；`dispatch_snippet` 把可调度
  模块清单注入系统提示，让 Agent 自主唤起/建议切专精模式；调用面按模式契约强制裁剪。
- **底层目标**：一句自然语言 → 裁层 → 动词候选 → 一键执行，全程尊重当前模式的工具面；
  拒绝时如实回报并指路 `mode.set`（不臆造、不静默）。
- **现状**：MentionRouter + capability_manifest + 模式感知 capabilities/prompt 就位；
  `/api/session.invoke`/`open_app` 已按模式强裁（PR #34）；本轮补齐 `/api/route`
  同一契约（blocked_by_mode + hint），AI 交互面板同步显示拦截原因；实测
  `open notepad…` 落通用层且候选跨 app 命中。
- **现状追加**：语义检索加中英同义桥 + 动词名/别名直命中加权（`导出制造文件`→`export_gerbers`）；
  工程交接状态机 `core/handoff.py` 落地（`/api/project.create|advance|status|list`），
  阶段完工→交接指引（@句柄 + 建议 `mode.set domain:<app>`），持久化 JSON、拒绝如实回报。
- **缺口**：交接尚未与 dispatch_snippet 提示词面自动联动（活动工程注入系统提示）；
  语义桥为领域小词表，通用语义（向量/大词表）待演化。
