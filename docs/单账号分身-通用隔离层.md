# 单账号多分身 · 通用隔离层（三机制归一）

> 道法自然：不再"给某几个软件零散做隔离"，而是把本仓已落地的三条隔离机制归一成
> **一条由强到弱的档位阶梯 + 一个选档解析器**，从底层统一裁决"为某分身隔离运行某软件"。

## 档位阶梯（强 → 弱）

| 档位 | 机制 | 隔离保证 | 成本 |
|---|---|---|---|
| `account` | 独立 Windows 本地账号（`core/accounts`） | OS 级全隔离：文件系统/HKCU/全局命名对象/GPU | 高（建号管理） |
| `session` | 单账号多路 RDP 会话（rdpwrap，`desktop/tunnel` 租约） | 独立完整桌面栈/输入队列/GPU 合成——**单账号类多账号分离的本源突破** | 中（firstlogon 已自动化） |
| `desktop` | `CreateDesktop` 隔离桌面 + 消息级 I/O（`core/adapter/win_desktop`） | 窗口栈/输入焦点；**不隔离** per-user 全局互斥体、无 GPU 离屏合成 | 零配置 |
| `appdata` | per-clone user-data-dir（`core/clone/app_isolation`） | 收窄"有数据目录开关"软件的单实例锁到 per-clone | 零配置 |

## 软件需求画像

每个软件按单实例行为归类，得到**最低档位需求**：

- Electron/Chromium（vscode / devin-desktop / chrome / edge）→ 锁在 user-data-dir 内 → `appdata` 即够；
- 全局互斥体且无开关（office / wechat 类）→ 需 `session` 及以上；
- 必须真 GPU 合成 → 需 `session` 及以上；
- **未登记软件保守按全局互斥体处理**——绝不乐观假设能零配置隔离。

## 解析器（纯逻辑 · CI 可测）

```python
from core.clone import resolve_isolation, IsolationTier, ALL_TIERS

resolve_isolation("vscode", "session-2")              # 缺省零配置三档 → appdata，isolated=True
resolve_isolation("wechat", "c1")                      # 零配置 → isolated=False，如实说明缺口
resolve_isolation("wechat", "c1", ALL_TIERS)           # 环境全就绪 → session（最省成本）
resolve_isolation("wechat", "c1", ALL_TIERS, True)     # prefer_strongest → account
```

- 选档规则：在**可用档位**里挑 **≥ 软件最低需求** 的最省成本档（`prefer_strongest=True` 选最强）。
- 选不出即 `isolated=False` + 缺口说明（如"需装 rdpwrap 开多会话或建独立账号"），绝不假装能隔离。
- 凡登记了 user-data-dir 策略的软件，无论选中哪档都附带 per-clone `app_launch` 规格（双保险，
  偏好/扩展互不覆盖）。

## 控制面接入

- REST：`POST /api/clone.plan {app_id, clone_id, tiers?, prefer_strongest?}`、
  `POST /api/clone.matrix {app_ids, ...}`（`tiers` 取值 `none/appdata/desktop/session/account`）。
- MCP：`clone_plan`、`clone_matrix` 工具（`bridge/mcp.py`）。

## 与既有成果的关系

本层**不新造机制**，只做统一裁决：`appdata` 落地仍走 `dao-clone-open.ps1`；`session` 落地仍走
`desktop/tunnel` 租约（`account:<名>#<分身号>`）；`desktop` 落地仍走 `win_desktop` 适配器；
`account` 落地仍走 `/api/account.create`。诚实边界与 [单账号分身-应用层隔离](单账号分身-应用层隔离.md)
一脉相承并体系化。
